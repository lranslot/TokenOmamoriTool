## 1. 目的

Claude Code の開発環境を常駐監視し、トークン節約ツール群(RTK / ccusage / claude-mem)の
導入・効果測定・環境の健康チェックをワンストップで行う Windows デスクトップツール。

方針:
- AIの出力(回答)を削る方向の節約は行わない(対話の質を優先)
- ツール自体がトークンを消費する仕組み(API呼び出しでのカウント等)は使わない
- 3層最適化(I/O層・記憶層・ワークフロー層)のうち、I/O層 + 記憶層 + 監視 をカバーする

## 2. 想定環境

- 言語:C#
- 形式:GUIアプリ(WPF)、常駐型
- 表示:画面右下に小型ウィンドウ(オーバーレイ or タスクトレイ常駐 + ポップアップ)
- 実行環境:Windows
- 開発:Claude Code に本仕様書を渡して実装させる

## 3. カバーする最適化層

| 層 | 状態 | 担当 |
|---|---|---|
| I/O層(コマンド出力の圧縮) | ○ | RTK |
| 記憶層(セッションまたぎの記憶) | ○ | claude-mem(省トークン設定込み) |
| ワークフロー層(SKILL管理) | × 対象外 | 個人の作業スタイル依存のため自動導入しない |
| 測定・監視(層の土台) | ○ | ccusage + 本ツールの監視機能 |

## 4. 機能一覧

### 4.1 CLAUDE.md 監視
- 監視対象:プロジェクトの CLAUDE.md(パスは設定で指定、複数プロジェクト対応)
- 判定基準:200行以下、ファイルサイズ25KB以下
- 超過時:右下ウィンドウに警告表示(「CLAUDE.mdが肥大化しています:現在xxx行/xxKB」)
- 内容の推奨事項(参考表示):
  - やってほしいこと
  - プログラムの仕様
  - 守ってほしいルール

### 4.2 セッション肥大化の監視
- 監視対象:Claude Code のセッションログ(保存場所・形式は §6-1 参照)
- ファイルサイズ・更新状況から「セッションが長引いている」ことを検知
- しきい値超過時:「/compact の実行を推奨」を右下ウィンドウに通知
- しきい値は設定ファイルで調整可能にする

### 4.3 ツールのワンクリック導入(セットアップウィザード)
以下を順番にチェックし、未導入のものをワンクリックで自動導入する。

#### (a) 前提:Rust ツールチェーン
- rustup / cargo の有無を確認(which/where cargo)
- 未導入なら rustup 公式インストーラを取得して導入
- ※ RTKのバイナリ配布を使う場合は不要になる(§6-5 方法B参照)

#### (b) RTK(必須)★重要な注意点あり
- 導入コマンド:cargo install --git https://github.com/rtk-ai/rtk
- 【必須】--git フラグを必ず使うこと。
  crates.io に同名の別パッケージ(Rust Type Kit)が存在するため、
  `cargo install rtk` では別物がインストールされてしまう。
- 導入確認:rtk --version 実行。さらに rtk gain が動作するかで正しいパッケージか判定
  (rtk gain が command not found → 別パッケージ混入の可能性 → 警告表示)
- フック設定:rtk init -g を実行(Claude Code の PreToolUse フックが設定される)
- 設定後「Claude Code の再起動が必要」の旨をユーザーに表示

#### (c) ccusage(必須:測定用)
- トークン消費の測定・ベースライン確認用
- npm 系ツールのため Node.js の有無を確認し、未導入なら導入を案内(または自動導入)
- 導入方法・出力形式は §6-4 参照

#### (d) claude-mem(導入する:記憶層)
- Claude Code のプラグイン機構経由で導入(CLI自動化の可否は §6-2 参照)
- 【重要】導入後、省トークン設定を自動適用する:
  - 設定ファイル:~/.claude-mem/settings.json
  - 方針:「事前の自動注入を極限まで絞り、必要な情報は都度取得」
  - CONTEXT_OBSERVATIONS=1(インデックス注入は直近1件のみ)
  - CONTEXT_FULL_COUNT=0(全文展開ゼロ、詳細は都度取得)
  - CONTEXT_SESSION_COUNT=1(直近1セッションの文脈のみ保持)
  - 理由:デフォルト設定のままだと観測記録の蓄積とともにセッション開始時の
    自動注入でトークン消費が膨らむ既知の問題があるため

#### 導入しないもの(明記)
- Caveman(出力を削るツール → 方針に反する)
- kizami(claude-mem と同カテゴリで重複)
- Token Optimizer / Token Savior(中〜大規模コードベース向け。現段階では旨味が薄い)
- Context Mode(小規模では逆効果の報告あり)

### 4.4 削減効果の表示(右下ウィンドウ)
- rtk gain の出力をパースして表示:累計削減トークン数、削減率
- rtk gain --history で直近のコマンド別削減も取得可能(詳細画面で表示)
- rtk discover の結果(まだ最適化できていないコマンド)も参考表示できると良い
- ccusage の結果表示(可能なら統合、難しければボタンでレポート起動)
- 問題チェック:CLAUDE.md超過・セッション肥大・RTKフック未設定などを
  アイコン(○/△/×)で一覧表示

### 4.5 他AI対応(設計レベルで対応、v1はClaude決め打ち)
- 設定ファイル(config.json 等)にAIごとの差分を追い出す設計にする:

```json
{
  "targetAI": "claude",
  "instructionFile": { "path": "CLAUDE.md", "maxLines": 200, "maxSizeKB": 25 },
  "sessionLogPath": "~/.claude/projects/",
  "tools": ["rtk", "ccusage", "claude-mem"]
}
```

- 部品ごとの他AI互換性:
  - RTK → 対応可能(rtk init -g --gemini / --codex / --agent cursor 等の公式オプションあり)
  - claude-mem → Claude専用(Claude Codeのライフサイクルフック依存)
  - ccusage → Claude Code向け
  - 指示ファイル監視 → パス・上限を設定化すれば流用可能(AGENTS.md等)
- v1 は targetAI=claude のみ実装。ただし上記の設定構造は最初から入れておく。

## 6. 調査結果(旧「要調査項目」・2026-07-05調べ)

旧仕様の要調査事項はすべて調査済み。以下が確定情報。

### 6-1. Claude Code セッションログの保存場所とフォーマット【解決】

- 保存場所:`~/.claude/projects/<エンコード済みプロジェクトパス>/*.jsonl`
  - Windowsでは `~/.claude` は `%USERPROFILE%\.claude` に解決される
  - エンコード規則:プロジェクトパスの `/`(または `\`)を `-` に置換し先頭に `-` を付ける
    例: `/Users/foo/dev/my-project` → `-Users-foo-dev-my-project`
  - `agent-` で始まるファイルはサブエージェントの履歴(監視対象から除外してよい)
- 形式:JSONL(1行1イベントのJSON)
  - 各行に type(user / assistant 等)、message、timestamp、sessionId、uuid などのフィールド
- 注意点:
  - 環境変数 `CLAUDE_CONFIG_DIR` が設定されていると場所が変わる(設定で対応できるようにする)
  - `CLAUDE_CODE_SKIP_PROMPT_HISTORY=1` だとログ自体が書かれない
- 監視実装:対象プロジェクトのフォルダ内で最新更新のjsonlのファイルサイズを
  FileSystemWatcher等で監視すればセッション肥大の検知に使える

### 6-2. claude-mem のCLI自動導入【ほぼ解決・要実機確認】

- 通常導入は Claude Code 内の `/plugin marketplace add` → `/plugin install` の2コマンド
- Claude Code には CLI からの `claude plugin install <名前>` コマンドが存在する
  (例:claude plugin install superpowers の実績記事あり)
  → claude-mem も同様に CLI から自動化できる可能性が高い。実装時に実機で確認
- 注意:`npm install -g claude-mem` はSDK/ライブラリのみでプラグインとしては機能しない
  ので、この方法は使わないこと
- 導入後の省トークン設定:`~/.claude-mem/settings.json` を本ツールから直接書き換え可能
  (CONTEXT_OBSERVATIONS=1, CONTEXT_FULL_COUNT=0, CONTEXT_SESSION_COUNT=1)

### 6-3. rtk gain の出力フォーマット【概ね判明】

- 出力例(実測記事より):
  ```
  Total commands: 998
  Tokens saved: 2.2M (92.3%)
  Total exec time: 23m22s (avg 1.4s)
  ```
- `rtk gain --graph` で30日間のASCIIグラフ、`--history` でコマンド別履歴も取得可能
- 節約データはローカルのSQLiteに保存されている
- 実装方針:まず `rtk gain` の標準出力を正規表現でパース。
  バージョンによる表記ゆれがあり得るので、パース失敗時は生テキストをそのまま表示に
  フォールバックする設計にする

### 6-4. ccusage の導入方法と出力形式【解決】

- 推奨はグローバルインストールではなく都度実行:`npx ccusage@latest`
  (常駐しないCLIで、価格データも最新が適用されるため公式も@latest付きを推奨)
  → 本ツールでは「導入」ではなく「Node.jsの確認 + 実行ラッパー」として実装すればよい
- JSON出力対応:`npx ccusage@latest daily --json` でプログラム処理しやすいJSONが取れる
  → C#側は JsonSerializer でそのままパース可能。表示統合が容易
- 主なサブコマンド:daily / weekly / monthly / session / blocks(5時間課金ブロック)
- ccusage も ~/.claude/projects/ のJSONLを読むだけで外部送信なし(オフライン動作)
- 前提:Node.js 20+ が必要。未導入なら導入を案内する

### 6-5. Windows での rustup 自動インストール【解決・ただし代替案あり】

- 方法A(従来案):rustup-init.exe をダウンロードして `-y` フラグでサイレント実行
  または `winget install Rustlang.Rustup`(Windows 10/11標準のwingetを利用)
- 方法B(推奨):RTKは公式ビルド済みWindowsバイナリを配布している
  → GitHub Releases からバイナリを直接ダウンロードすれば
    Rustツールチェーン自体が不要になり、セットアップが大幅に簡略化できる
- 実装方針:方法Bを第一候補とし、バイナリ配布が使えない場合のフォールバックとして
  方法A(rustup + cargo install --git)を実装する
- 方法Aを使う場合の再確認:cargo install は必ず
  `cargo install --git https://github.com/rtk-ai/rtk` とすること(同名パッケージ対策)

### 6-6. 実装順序(実績:この順で完了)

1. 監視機能(6-1のパスを見るだけ。外部依存なし)→ 最初に動くものができる
2. rtk gain / ccusage --json のパース・表示(読み取りのみで安全)
3. インストール/アンインストールUI(右クリックメニュー方式で実装)

## 8. トレイ常駐(v0.3追加)

### 8.1 基本方針
- 小窓(右下ステータスウィンドウ)を既定の表示形態として維持する。トレイ常駐は
  小窓を「隠す」ための手段であり、置き換えではない。
- ライブラリは `H.NotifyIcon.Wpf`(NuGet)を使用する。WinForms 参照の混在は不可。

### 8.2 トレイへの格納と終了
- ×ボタン:アプリを終了せず、小窓を非表示にしてトレイへ格納する。
- アプリの終了は、右クリックメニュー(ウィンドウ・トレイ共通)末尾の「終了」から行える。
  どちらの「終了」もトレイ格納ではなく完全終了。
- 初回格納時のみ、バルーン通知(またはトースト)で
  「トレイに常駐しています。終了はトレイアイコンの右クリックから行えます」と案内する。
  「初回のみ」の判定は settings.json に既出フラグ(例: `tray.firstHideNoticeShown`)を
  保存して行う。

### 8.3 トレイアイコンの操作
- 左クリック:小窓の表示/非表示をトグルする。表示時は従来どおり画面右下に固定。
- 右クリック:メニューを表示する。項目は MainWindow の ContextMenu と完全に同一構成
  (CLAUDE.mdダイエット指示をコピー → CLAUDE.mdのフォルダを開く → /compactをコピー →
  ツール管理 → 設定 → 説明 → セパレータ → 終了)。メニュー(ウィンドウ・トレイ共通)の
  末尾に「終了」を含む。有効/無効・警告色・サブメニューの動的切替ロジックは既存実装を
  共有すること(コピペで二重管理しない。共通のメニュー構築処理に切り出す)。

### 8.4 ツールチップ
- トレイアイコンのホバー時ツールチップに健康状態サマリーを表示する。
  例:「CLAUDE.md 45% / セッション 62% / 本日節約 12.3K」
- 値は既存のポーリング結果を流用し、ポーリングのたびに更新する。
- 表示できない値(ccusage未導入等)は該当項目を省略する。

### 8.5 アイコンの状態表示
- アイコンは2状態:正常/警告。
- CLAUDE.md またはセッションログのいずれかが 70% 以上になったら警告アイコンに
  切り替える。両方が 70% 未満に戻ったら正常アイコンに戻す。
- 点滅・アニメーションは行わない。

### 8.6 警告トースト
- 発火条件:「70%未満 → 70%以上」への遷移を検知した瞬間のみ
  (エッジトリガー。超過が継続している間は再通知しない。
  `CompactBoundaryTracker` と同様の境界検知の考え方)。
- CLAUDE.md とセッションログはそれぞれ独立に判定する。
- トーストのクリックで小窓を表示状態に復帰させる。
- トーストは `Microsoft.Toolkit.Uwp.Notifications` を使用する。
- 一度 70% を下回ってから再度超えた場合は、再度通知してよい。
- 初回観測時点ですでに70%以上の場合は発火しない(遷移が観測されていないため。
  アイコン・バーの警告表示で代替)。

### 8.7 設定
settings.json に以下を追加する。設定ウィンドウにも対応するUIを追加する。

| 設定ウィンドウの項目 | settings.json のキー | 既定値 | 意味 |
| --- | --- | --- | --- |
| ×ボタンでトレイに格納する | `tray.hideOnClose` | true | false の場合、×で従来どおり終了 |
| 警告をトースト通知する | `tray.toastOnWarning` | true | false の場合、アイコン色変化のみ |

- `tray.firstHideNoticeShown`(既定 false)は内部フラグであり、設定ウィンドウには
  表示しない。

## 9. カンパ導線(v0.3追加)

### 9.1 基本方針
- 押し売りをしない。価値(節約実績)を実感した瞬間に一度だけ提案する。
- 常設導線は目立たない場所に2つだけ置く。

### 9.2 マイルストーン提案
- RTK の累計節約トークン数(既存の `rtk gain` パース結果)がマイルストーンに
  到達したとき、小窓内に控えめなバナーを1回だけ表示する。
- マイルストーン:10万 / 100万 / 1000万 トークン。
- バナー文言例:「累計100万トークン節約しました 🎉 役に立っていたら開発を
  応援してもらえると嬉しいです」+「応援する」ボタン(カンパページを既定ブラウザで
  開く)+「閉じる」ボタン。
- 閉じたマイルストーンは二度と表示しない。表示済みマイルストーンは settings.json
  (例: `donation.shownMilestones` の配列)に記録する。
- トーストでは通知しない(小窓内バナーのみ)。
- バナー表示中も小窓の幅300px固定は維持する(高さ方向のみ一時的に拡張してよい。
  バナーを閉じたら元に戻す)。

### 9.3 常設導線
- 右クリックメニュー(MainWindow・トレイ共通)の「説明」の下に
  「☕ 開発を応援」を追加する。クリックでカンパページを既定ブラウザで開く。
- ヘルプウィンドウ内にも同様のリンクを1つ置く。
- カンパページのURLは settings.json ではなく定数(1箇所)で管理する
  (`DonationMilestones.DonatePageUrl`)。
  ※ 確定済み:https://ko-fi.com/lranslot(反映済み)

### 9.4 実装上の注意
- カンパ機能は監視・節約機能から完全に独立させる。カンパ関連のコードが例外を
  出しても監視機能に影響しないこと。
- 通信は行わない(URLをブラウザで開くだけ)。ツール自体がトークンも通信も
  消費しない方針は維持する。

## 10. テスト方針(トレイ・カンパ分)

- マイルストーン判定(未達→到達の検知、表示済み除外)は純粋ロジックとして
  `Services/` に切り出し、単体テストの対象とする。
- 警告トーストのエッジトリガー判定(70%境界の遷移検知)も同様に純粋ロジックとして
  切り出してテストする。
- トレイアイコン・トースト・バナーUIそのものは既存方針どおりテスト対象外。

## 11. 多言語対応(日本語 / English)(v0.3追補)

### 11.1 基本方針
- v1 から日本語・英語の2言語に対応する。
- UI に表示するすべての文字列(ウィンドウ、メニュー、ツールチップ、トースト、
  バナー、設定項目ラベル、ヘルプ本文)をハードコードせず、言語リソースに分離する。
- クリップボードにコピーする指示文(CLAUDE.mdダイエット指示)も言語リソース化する。
  ただし `/compact` などのコマンド文字列自体は言語によらず不変。

### 11.2 実装方式
- 言語別 ResourceDictionary(例: `Resources/Strings.ja.xaml` /
  `Resources/Strings.en.xaml`)+ XAML からは DynamicResource 参照とし、
  言語切替を再起動なしで即時反映する。
- コードビハインドから文字列を取得する箇所は、共通のローカライズサービス
  (例: `Services/LocalizationService`)経由で取得する。切替時に更新イベントを
  発行し、動的に組み立てるメニュー・ツールチップも再構築する。
- 数値・日付の書式は文字列リソースとは分けて考え、既存表示を崩さないこと。

### 11.3 言語切替メニュー
- 右クリックメニュー(MainWindow・トレイ共通)の「設定」の直前に
  サブメニュー「Language / 言語」を追加する。ラベルは言語設定によらず常に
  この併記表記とする(誤って切り替えた際に戻れるようにするため)。
- サブメニュー項目:「日本語」「English」。現在選択中の言語にチェックマークを表示。
- 選択すると即時にUI全体へ反映し、settings.json に保存する。

### 11.4 既定言語と設定
- settings.json に `language` キーを追加(値: `"ja"` / `"en"`)。
- キーが存在しない初回起動時は、OSのUIカルチャ(`CultureInfo.CurrentUICulture`)が
  日本語なら `ja`、それ以外は `en` を既定とし、その値を保存する。
- 不正な値が入っていた場合は `en` にフォールバックする。

### 11.5 翻訳の品質基準
- 英語リソースは機械的な直訳ではなく、Windowsデスクトップアプリとして自然な
  UIライティングにする(例:「CLAUDE.mdダイエット指示をコピー」→
  "Copy CLAUDE.md diet instructions" など、簡潔な命令形)。
- アプリ表示名は言語別とする:日本語「トークン節約お守りツール」/
  英語 "Token Omamori Tool"(Omamori はローマ字のまま製品名として扱う)。
- 未翻訳キーがあった場合は英語にフォールバックし、起動時に Debug ログへ警告を
  出す(ユーザーには見せない)。

### 11.6 テスト方針(多言語分)
- 言語リソースの網羅チェック(ja/en 両辞書のキー集合が一致すること)を
  単体テストとして追加する。これによりキーの追加漏れをCIで検出できる。
- LocalizationService のフォールバック動作(不正値→en)もテスト対象とする。

### 11.7 README
- README は日英併記(先頭に English follows Japanese の注記、または上下2部構成)
  とする。

## 12. 外見(お守りモチーフ)(v0.3追補2)

### 12.1 基本方針
- アプリ全体を「お守り」モチーフで統一する。ただし本ツールは実用の監視ツールで
  あるため、装飾は「和の気配を足す」程度に抑え、情報の視認性を最優先とする。
- 色は一元管理する。ステータス色・背景色などは共通の ResourceDictionary
  (例: `Resources/Theme.xaml`)に定義し、XAML・アイコン描画コードの双方から
  参照する。個別画面へのハードコード禁止。

### 12.2 カラーパレット
| 名前(リソースキー例) | 用途 | 色の方向性 |
| --- | --- | --- |
| `OmamoriHealthy` | 正常状態 | 常盤色(深い緑、例 #1B813E 系) |
| `OmamoriWarning` | 警告状態(70%以上) | 琥珀色(例 #CA7A2C 系) |
| `OmamoriDanger` | 危険状態(90%以上、バー・メニューのみ) | 紅緋(例 #C73E3A 系)。トレイ・ミニお守りは2状態のまま |
| `OmamoriAccent` | 紐・縁・アクセント | 金茶〜白系(アイコンの紐や縁取り) |
| `WashiBackground` | 小窓・サブウィンドウの背景 | 生成り色(和紙風オフホワイト、例 #F8F4E6 系) |

- 具体的な色値は実装時に上記の方向性の範囲で調整してよい(コントラスト比を優先)。
- 既存の警告色(バー・メニューの警告表示)も `OmamoriWarning` に統一する。
- ※ 実装済みの色一覧:常盤#1B813E / 琥珀#CA7A2C / 紅緋#C73E3A / 金茶#B98C4A /
  生成り#F8F4E6(OmamoriTheme.cs / Theme.xaml の二段構えで一元管理)

### 12.3 トレイアイコン
- 図形:お守り袋のシルエット(上が尖った五角形寄りの袋+上部に小さな紐の輪)。
  16×16 で判別できるよう、紐の結び目などの細部は省略した抽象形とする。
- 状態表現:袋全体の塗り色をステータス色とする。
  正常= `OmamoriHealthy`、警告= `OmamoriWarning`。2状態、点滅なし(§8.5 を継承)。
- 実装:共有図形定義(1箇所、アプリアイコンと共通)から生成した2状態の .ico を
  URI 参照で切り替える(`GeneratedIconSource` は文字+楕円/矩形のみで任意図形を
  描画できないため、動的描画は使わない)。

### 12.4 アプリアイコン(.ico)
- EXE・タスクバー・ウィンドウ左上・Alt+Tab に使うアプリアイコンを作成する。
- 図形:トレイ用の抽象形をベースに、大サイズでは紐の結び(叶結び風の簡略表現)
  まで描いた「ちゃんとしたお守り」とする。袋の地色は `OmamoriHealthy`(緑)固定
  (アプリアイコンは状態で変化しない)。
- 形式:マルチサイズ .ico(16/32/48/256 を含む)。SVG またはコードで原版を管理し、
  生成手順(スクリプトまたはコマンド)をリポジトリに残す。
- csproj の `ApplicationIcon` と各 Window の `Icon` に設定する。

### 12.5 小窓(メインウィンドウ)の外見調整
- 背景を `WashiBackground` にし、角を軽く丸める(CornerRadius 6〜8 程度)。
- タイトル部の左に小さなお守りアイコン(トレイと同じ抽象形・ステータス色連動)を
  表示する。トレイに気づいていないユーザーにも状態が伝わる。
- 幅300px固定・レイアウト構成・フォントは変更しない。
- バー・警告文の警告色は §12.2 のパレットに置き換える。
- サブウィンドウ(設定・説明・ツール操作)は背景色の統一のみ行い、
  レイアウトは変更しない。

### 12.6 やらないこと
- アニメーション・点滅・グラデーションの多用
- フォント変更、和柄テクスチャなど可読性を下げる装飾
- ウィンドウの透過・変形ウィンドウ(実装コストと不具合リスクに見合わない)

### 12.7 テスト方針(外見分)
- 描画・アイコンは既存方針どおり自動テスト対象外。
- ステータス→色の対応判定をロジックとして切り出している場合(既存の
  ワースト使用率計算等)は既存テストを維持する。色値そのものはテストしない。

## 13. EXE化・配布(v0.3追補3)

### 13.1 基本方針
- 配布形態は「ポータブル型」とする:インストーラーなし、ZIPを展開して EXE を
  実行するだけ。settings.json は従来どおり EXE と同じディレクトリに保存する
  (この設計はポータブル配布と相性が良いので変更しない)。
- 配布物は self-contained・単一ファイル EXE を第一とする(.NETランタイム不要)。
  サイズが大きくなる(150MB前後)ことは許容する。

### 13.2 発行(publish)設定
- `tools/publish.ps1` として発行スクリプトをリポジトリに追加する。内容:
  - `dotnet publish` / Release / `win-x64` / `--self-contained true`
  - `PublishSingleFile=true`、`IncludeNativeLibrariesForSelfExtraction=true`
  - WPF のため **Trimming(PublishTrimmed)と ReadyToRun の圧縮系オプションは
    使用しない**(WPFはトリミング非対応。壊れる)
  - 出力を `publish/` 配下にまとめ、`TokenOmamoriTool-vX.Y.Z-win-x64.zip` に圧縮
  - ZIP の SHA256 を計算してテキストファイルに出力
- 発行後の EXE で以下を必ずスモークテストする(単一ファイル化で壊れやすい点):
  - 起動し、アイコン(ウィンドウ・タスクバー・トレイ)が正しく表示される
  - ja/en 切替が動く(ResourceDictionary の pack URI 解決の確認)
  - settings.json が EXE と同じディレクトリに生成される
  - トレイ格納・復帰・終了が動く

### 13.3 バージョン管理
- csproj に `Version`(例 1.0.0)を設定し、以後リリースごとに更新する。
- ヘルプウィンドウにバージョン番号を表示する(アセンブリ情報から取得。
  ハードコード禁止)。
- Git のタグ `vX.Y.Z` とリリースZIP名・csproj の Version を一致させる。

### 13.4 GitHub Releases での配布
- リリースごとに添付するもの:
  - `TokenOmamoriTool-vX.Y.Z-win-x64.zip`(EXE + 同梱README的なテキストは不要、
    リポジトリREADMEへ誘導)
  - SHA256 チェックサム
- リリースノートは日英併記(READMEと同じ「日本語先・English後」の順)。
- リリースノートのテンプレート(変更点・動作環境・既知の問題)を
  `docs/release_note_template.md` として用意する。

### 13.5 README への追記(配布関連)
README の日英両方に以下を追記する:

1. **インストール手順**: GitHub Releases から ZIP をダウンロード → 書き込み可能な
   フォルダ(例: ドキュメント配下や任意の作業フォルダ)に展開 → EXE を実行。
   `Program Files` 配下は settings.json が書けないため非推奨と明記。
2. **SmartScreen の説明**: 本アプリはコード署名をしていないため、初回実行時に
   Windows SmartScreen の警告が出る。「詳細情報」→「実行」で起動できる旨と、
   ソースが公開されておりビルドも再現可能である旨を記載する。
3. **セキュリティソフトの誤検知**: self-contained 単一ファイル EXE は
   まれに誤検知されることがある。その場合の対処(除外設定 or 自前ビルド)を
   一言案内する。
4. **RTKに関する既知の制約(Windows)**: Claude Code がコマンドを PowerShell
   ツール経由で実行した場合、RTK のフック(Bashツールのみ対応)は介入できず
   節約が伸びないことがある。また `rtk discover` が Windows でセッションを
   検出しない事象を確認している。これらは RTK 側の制約であり本ツールの
   表示は正確である旨を記載する(トラブルシュート節)。

### 13.6 やらないこと(v1)
- コード署名(証明書の費用対効果が個人開発の初期段階に見合わない。
  将来の検討課題として README に含めない)
- インストーラー(MSI/MSIX)化
- 自動アップデート機構
- winget / Microsoft Store への登録(反応を見てから検討)

### 13.7 テスト方針(配布分)
- publish.ps1 の実行と §13.2 のスモークテストは手動で行う(自動テスト対象外)。
- バージョン表示がアセンブリ情報から取得されていることは、可能なら
  ロジックを切り出して単体テストしてよい(必須ではない)。
