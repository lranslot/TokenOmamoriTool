# 実装メモ：アーキテクチャ詳細

このファイルは `CLAUDE.md` の「アーキテクチャ注記」を肥大化させないための詳細置き場です。
仕様そのものは `docs/仕様書_統合版_v1.0.md` が authoritative。ここは実装済みコードの内部挙動・実機確認済みの細部・既知のはまりどころをまとめた
実装メモで、`CLAUDE.md` から参照される。

## セッションログ監視

Claude Code session logs live at `%USERPROFILE%\.claude\projects\<encoded-project-path>\*.jsonl`
(path-encoding rule: replace `/` or `\` with `-`, prefix with `-`). Respect `CLAUDE_CONFIG_DIR` env
var if set. Files prefixed `agent-` are sub-agent histories and should be excluded from monitoring.
Detect session bloat via file size/mtime (e.g. `FileSystemWatcher`), not by parsing every line.

## `/compact` 検出

jsonl files are append-only — running `/compact` does not shrink the file, so raw file size alone
can't reflect post-compact bloat. Claude Code writes a line containing `"isCompactSummary":true`
right after a compact. `Services/CompactBoundaryTracker` finds the byte offset just after the most
recent such line and reports bytes appended since (falling back to full file size if no compact has
happened yet). It caches the last-scanned offset per file path and only re-scans newly appended bytes
on subsequent calls, backing up to the nearest line boundary first (in case that boundary sits inside
a large line — real compact-summary lines have been observed at 25-30KB) — this is the one place
session log monitoring reads file *content*, not just size/mtime.

## CLAUDE.md ヘルスチェック

Flag when a project's CLAUDE.md exceeds 200 lines or 25KB (see `Services/ClaudeMdMonitor`).

## RTK 統合

Critical gotcha: install must use `cargo install --git https://github.com/rtk-ai/rtk` (never plain
`cargo install rtk`, which installs an unrelated crates.io package of the same name). Verify a correct
install by checking that `rtk gain` runs (not just `rtk --version`). Prefer RTK's prebuilt Windows
binary releases over the rustup+cargo path when possible. After `rtk init -g`, the user must restart
Claude Code for the PreToolUse hook to take effect. `rtk gain` output is plain text (`Total commands:`,
`Tokens saved: X (Y%)`, `Total exec time:`) — parse with regex and fall back to raw text display on
parse failure, since formatting may vary by version.

## RTK 本日分/累計の内訳表示（実装済み）

`rtk gain` は生の整数を返さず、常に自前で丸めた文字列（例: `2.2M`）しか出力しない。そのため「本日分」を
出すには、この丸め済み文字列を`Services/TokenCountFormatter.TryParse`で概算の数値に戻し、当日の基準点
との差分を取ってから同じ丸め規則(`TokenCountFormatter.Format`)で再度文字列化するしかない —
`rtk`側のDB/出力そのものを書き換えることは一切しない。累計欄は`rtk gain`の生文字列をそのまま使う
（自前で再丸めしない）。

基準点は`%LOCALAPPDATA%\TokenOmamoriTool\rtk-daily-baseline.json`に`{date, baselineTokensSaved}`を
保存する内部状態（`Services/RtkDailyBaselineTracker`）で、ユーザー向け`settings.json`とは別。
`ResolveBaseline(stored, currentCumulative, today)`は純粋関数：保存済みの`date`が今日と一致すればそのまま
維持、不一致なら「今の累計値」を新しい基準点として上書きする。日付切り替えは3段構え：
①次の0:00ちょうどに発火するワンショット`DispatcherTimer`（`MainWindow.ScheduleMidnightRollover`。発火後は
自分自身を翌日0:00に再セット）、②60秒ごとのRTKポーリング（`RefreshExternalToolsAsync`）が呼ぶたびに
毎回`ResolveBaseline`で日付チェックが走るので、スリープ復帰やタイマー取りこぼしがあっても次のポーリングで
必ず補足される、③起動直後の初回`RefreshExternalToolsAsync`呼び出しでも同じチェックが走るので、日付が
変わった翌朝に起動したケースも自動的に拾われる。基準点が累計を上回る（RTK側のカウンタが外部要因で
リセットされた等）場合は本日分を0にクランプし、マイナス表示にはしない。表示形式:
「節約(RTK)：本日 12.3K / 累計 456K tokens」。RTK未導入時はこのロジック自体を通らず、従来の
「節約(RTK)：未導入」のまま（内部状態ファイルも作られない）。

## Gain/usage 表示（実装済み）

`Services/ExternalCommandRunner` is the shared process runner both `RtkGainRunner` and `CcusageRunner`
sit on. It catches `Win32Exception` from `Process.Start` to mean "executable not found" — this is how
"節約(RTK)：未導入" / "本日消費：Node.js未導入" are detected, without the app ever crashing when those
tools are missing. `RtkGainParser.TryParse` and `CcusageParser.TryParseToday` are pure and
unit-tested; the `*Runner` classes that actually spawn processes are not (consistent with not
unit-testing the WPF UI — verify those by running the app). Gotcha: on Windows, `npx` (and `claude`)
are `.cmd` wrappers, so `Process.Start("npx", ...)` with `UseShellExecute=false` fails to resolve them
(CreateProcess doesn't consult PATHEXT) — `CcusageRunner` and the claude-mem services shell out via
`cmd.exe /c npx ...` / `cmd.exe /c claude ...` instead, and use the wrapped command's own exit code
(not `Started`) to detect "not installed". `rtk` is a real `.exe` (cargo/binary install) so it's
started directly. Both RTK/ccusage polling and claude-mem status polling run on their own
`DispatcherTimer` (default 60s, `ExternalTools.PollIntervalSeconds` in settings), separate from the 5s
monitoring timer, and `await` the process asynchronously so the UI thread never blocks on `npx`/`claude`
(which can take several seconds, longer on first run).

## claude-mem 統合

Install via Claude Code's plugin mechanism. Do NOT use `npm install -g claude-mem` (SDK-only, not a
working plugin). After install, this tool directly rewrites `~/.claude-mem/settings.json` to force a
low-token config: `CONTEXT_OBSERVATIONS=1`, `CONTEXT_FULL_COUNT=0`, `CONTEXT_SESSION_COUNT=1`.

## Install/uninstall UI（実装済み）— 右クリックContextMenu、setup-wizardボタンなし

`MainWindow` builds a fresh `ContextMenu` on every `ContextMenuOpening` (see
`Window_ContextMenuOpening`/`BuildContextMenu`) rather than mutating a persisted one, so its
「ツール管理」submenu labels (install vs. uninstall) always reflect the last 60s poll's
`ExternalToolsHealth.RtkInstalled` / `ClaudeMemState` without blocking the click on a fresh process
check. A placeholder `ContextMenu = new ContextMenu();` must be assigned in the constructor before
`ContextMenuOpening` will ever fire, and the handler must be attached to the `Window` (not an inner
`Panel` without a `Background`) or right-clicks on empty space won't be hit-test visible.
Installers/uninstallers are expressed as ordered `Models/OperationStep` lists (Description +
CommandPreview shown before running + async `Run`), executed one at a time by the generic
`ToolOperationWindow`; on failure it stops and lets the user retry — completed steps are skipped on
retry, only the failed step onward re-runs. All exceptions are caught per-step so a broken
install/uninstall can never take down the 5s/60s monitoring timers.

### RTK（`Services/RtkInstaller`/`RtkUninstaller`）

Binary-first — GitHub Releases `GET https://api.github.com/repos/rtk-ai/rtk/releases/latest`, find
the asset containing `pc-windows-msvc` ending in `.zip` (confirmed real asset name:
`rtk-x86_64-pc-windows-msvc.zip`), extract to `%LOCALAPPDATA%\TokenOmamoriTool\tools\rtk`, add that
dir to the **user** PATH env var (`EnvironmentVariableTarget.User`) *and* the current process's PATH
(so verification steps in the same run can find `rtk` immediately without an app restart). Only falls
back to `cargo install --git https://github.com/rtk-ai/rtk` (never without `--git`) if no matching
asset is found; installs rustup silently (`rustup-init.exe -y` from `https://win.rustup.rs/x86_64`)
first if `cargo` itself is missing. Verifies with `rtk --version` *and* `rtk gain` (matches the spec's
"impostor crates.io package" concern). The final `rtk init -g` hook-setup step runs as
`rtk init -g --auto-patch` with `120s` timeout (longer than the other rtk steps) and writes `"y\n"` to
the process's stdin — confirmed on a real machine that plain `rtk init -g` hangs on a
`Patch existing ...settings.json? [y/N]` confirmation prompt and times out; `--auto-patch` is RTK's
own documented non-interactive flag (README: "Non-interactive (CI/CD)"), and the stdin write is a
defensive second layer via `ExternalCommandRunner`'s optional `standardInput` parameter in case some
prompt still slips through. If it still times out (`Result.StandardError == "timeout"`), the failure
message tells the user to run `rtk init -g` manually in a terminal and answer `y` to the prompt.
Uninstall uses RTK's own `rtk init -g --uninstall` (confirmed
via README: removes the hook/RTK.md/settings.json entry — no need to hand-parse Claude Code's
settings.json), then `cargo uninstall rtk` or a plain directory delete depending on which method was
used — tracked in a small internal `%LOCALAPPDATA%\TokenOmamoriTool\tool-install-state.json`
(separate from the user-facing monitoring `settings.json`).

### claude-mem（`Services/ClaudeMemInstaller`/`ClaudeMemUninstaller`/`ClaudeMemStatusChecker`）

The `claude` CLI supports plugin management directly (`claude plugin list` /
`marketplace add <source>` / `install <plugin>` / `uninstall <plugin>` — confirmed by running
`claude plugin --help` on a real machine; this fully resolves spec doc 6-2's「要実機確認」). The real
marketplace is **`thedotmack/claude-mem`** (confirmed via its GitHub repo and docs.claude-mem.ai):
`claude plugin marketplace add thedotmack/claude-mem` then `claude plugin install claude-mem`. Detect
"installed" by checking whether `claude plugin list` output contains `claude-mem`; if `claude` itself
isn't on PATH (or its exit code is non-zero), the menu shows a disabled「claude CLIが見つかりません」
item instead of guessing. The exact schema of `~/.claude-mem/settings.json` isn't publicly documented,
so the 3 low-token keys are merged in via `JsonNode` (not a typed POCO) to avoid clobbering unknown
existing keys, and any existing file is copied to `settings.json.bak.<timestamp>` before being
overwritten. Uninstall only runs `claude plugin uninstall claude-mem` — `~/.claude-mem`'s data
directory is deliberately left alone (it's the user's memory data, not disposable cache).

### 設定/説明ウィンドウ

`SettingsWindow` mutates the *same* `AppSettings` instance `MainWindow` holds (not a copy) and calls
`SettingsLoader.Save`, so changes take effect immediately via an `onSaved` callback that resets both
`DispatcherTimer.Interval`s and calls `RefreshHealth()` — no file-watcher, no restart needed.
`HelpWindow` is static content (Q&A形式：「これは何？」＋「どうすればいい？」の2部構成)。

### CLAUDE.mdダイエット指示（右クリックメニュー）

「CLAUDE.mdダイエット指示をコピー」は状態連動：使用率（行数・サイズの大きい方）が70%未満（緑）なら
`IsEnabled = false` でグレーアウト。70%以上90%未満（黄）・90%以上（赤）でそれぞれ異なる文面
（`MainWindow.BuildClaudeMdDietMessage`）を実際の行数/サイズ/上限値を埋め込んで生成し、クリップボード
にコピーする。右クリックメニュー全体の項目文字色も、対応するバーの使用率に応じて
`UsageFractionToBrushConverter.BrushFor` と同じ閾値（0.9=赤、0.7=黄）で着色される
（`MainWindow.ApplyWarningColor`）。

## ccusage 統合

Don't install globally — shell out to `npx ccusage@latest` per the spec's findings (keeps pricing
data current). Use `npx ccusage@latest daily --json` (and weekly/monthly/session/blocks variants) for
machine-readable output via `JsonSerializer`. Requires Node.js 20+; check for it, don't assume.

**`@latest` の新版公開直後の恒久失敗ループ(2026-07-18 実機で発生・対策済み)**: ccusage の
新バージョンが npm に公開されると、次回の `npx ccusage@latest` はまずダウンロードを行う。
このダウンロードが旧タイムアウト45秒を超えると `ExternalCommandRunner` がプロセスツリーごと
kill し、npx キャッシュが未完成のまま残る → 次のポーリングでも再ダウンロードから始まり再び
45秒で kill → キャッシュが永遠に完成せず「本日消費:実行に失敗しました」が出続ける。対策:

1. ccusage 実行のみタイムアウトを **180秒** に延長(`CcusageRunner`。rtk 等の他コマンドは従来どおり)。
2. `Services/LastGoodDisplayCache`(純粋クラス・単体テスト有): RTK/ccusage の表示行は
   実行/パース失敗時も同日の前回成功値を「〜(更新失敗)」付きで表示し続ける。メモリ内保持のみ
   (再起動で消える)。日付が変わったら破棄し、前日の値を今日の値として見せない。
   `MainWindow.RefreshExternalToolsAsync` が `ParsedOk` を成功判定として両行に適用。

## UI shape

Small status window anchored bottom-right (300px wide), showing a progress-bar checklist for
CLAUDE.md size / session bloat, cumulative tokens-saved from `rtk gain` (labeled「節約(RTK)：」), and
ccusage's today total (labeled「本日消費：」). **No dedicated setup-wizard button** — install/uninstall
of RTK and claude-mem, opening CLAUDE.md, copying the diet instruction / `/compact`, settings, and
help are all reached via a right-click `ContextMenu` on the main window so the main window itself never
changes size. Bottom-right anchoring is done in code (`MainWindow.PositionBottomRight`, re-run on
`SizeChanged` because `SizeToContent=Height` lets the height vary per poll).

## トレイ常駐（実装済み、仕様追加v0.3 §8）

Library pins and TFM — do not "upgrade" these blindly:

- **`H.NotifyIcon.Wpf` is pinned to 2.3.2**: 2.4.x only ships `net462`/`net10.0-windows` assets, so
  a net8 project can't restore it. 2.3.2 is the newest version with a `net8.0-windows7.0` asset
  (verified via nuspec on nuget.org).
- **TFM changed from `net8.0-windows` to `net8.0-windows10.0.17763.0`** (both the app *and* the test
  project — a plain `net8.0-windows` project can't reference a 17763 one). Required by
  `Microsoft.Toolkit.Uwp.Notifications` 7.1.3, whose toast APIs (`ToastContentBuilder.Show()`) only
  exist in its `net5.0-windows10.0.17763` asset; on plain `net8.0-windows` the package silently
  resolves to the API-less `net5.0` asset.
- **`TaskbarIcon.IconSource` does not accept arbitrary `ImageSource`**: only URI-backed
  `BitmapImage`/`BitmapFrame` or H.NotifyIcon's own `GeneratedIconSource`. A `RenderTargetBitmap`
  throws `NotImplementedException` *asynchronously* inside `TaskbarIcon.OnIconSourceChanged` (fires
  on the dispatcher, so a try/catch around the assignment does NOT catch it — it crashed the app on
  a real machine). `TrayIconFactory` therefore builds the two state icons (green/amber ellipse,
  spec §8.5) as `GeneratedIconSource`.

Behavior wiring (all in `MainWindow`):

- `App.xaml` sets `ShutdownMode="OnExplicitShutdown"` so `Hide()` keeps the app alive;
  `MainWindow.OnClosed` calls `Application.Current.Shutdown()` when the window *really* closes.
  × hides to tray (`OnClosing` cancels the close) unless `tray.hideOnClose` is false, the exit came
  from a menu's「終了」(window or tray — both are a full quit, `_exitRequested`), or tray-icon
  creation failed at startup (then × falls back to a normal exit so the app can't get stranded with
  no window and no tray icon).
- The tray right-click menu is the same `BuildContextMenu()` the window uses — both menus are
  identical incl. the trailing separator +「終了」— rebuilt fresh on `TrayRightMouseDown`
  (TaskbarIcon shows the menu on right-mouse-*up*, so -down is early enough), mirroring the
  existing `ContextMenuOpening` rebuild pattern. The worst-project
  fraction computation moved out of `BuildContextMenu` into `UpdateWorstFractions()` (run by every
  `RefreshHealth`) because the tray icon state/tooltip and the toast edge detector need the same
  numbers.
- Left-click toggles the window (`TrayLeftMouseUp`); tooltip is rebuilt each poll from the stored
  fractions +`RtkDailyDisplayBuilder.BuildTodayText` (item omitted when RTK/parse unavailable).
- Warning toasts (spec §8.6) are edge-triggered by `Services/WarningEdgeDetector` (pure,
  unit-tested): fires only on a below→above-70% transition per key; the *first observation* of a key
  never fires (so an already-bloated project doesn't toast on every app launch); dropping below 70%
  re-arms. The detector is fed even while `tray.toastOnWarning` is off so its state stays current.
  Toast clicks restore the window via `ToastNotificationManagerCompat.OnActivated` (fires on a
  background thread — dispatch to the UI thread). All toast sends go through a catch-all so a
  notification failure can never take down the monitoring timers. The one-time「トレイに常駐して
  います」notice on first hide also uses a toast and persists `tray.firstHideNoticeShown` via
  `SettingsLoader.Save`.
- Settings: `tray.hideOnClose` /`tray.toastOnWarning` are user-facing checkboxes in
  `SettingsWindow`;`tray.firstHideNoticeShown` is internal and deliberately not reset by the
  設定ウィンドウ's「初期値に戻す」button.

## カンパ導線（実装済み、仕様追加v0.3 §9）

- URL は `Services/DonationMilestones.DonatePageUrl` の**定数1箇所**のみ（仕様 §9.3）。
  確定済み: `https://ko-fi.com/lranslot`（README の支援セクションも同URL）。通信は一切せず、
  既定ブラウザで開くだけ。
- マイルストーン判定は `Services/DonationMilestones`（純粋・単体テスト済み）:
  `FindUnshownMilestone` は「到達済みかつ未表示」の**最上位1件だけ**返す（初回計測でいきなり2M
  だった場合に3枚バナーが積まれないように）。`MarkShown(shown, M)` は M 以下を全部表示済みに
  する（上位を閉じた後に下位が今さら出ないように）。表示済みは settings.json の
  `donation.shownMilestones` に保存。
- 累計値は `rtk gain` の丸め文字列を `TokenCountFormatter.TryParse` で概算に戻したもの
  （`MainWindow._rtkCumulativeTokens`）。RTK未導入/パース失敗時は null → バナーは出ない。
- バナーは MainWindow 内の `DonationBanner`（既定 Collapsed）。幅300px固定は不変、
  `SizeToContent=Height` なので高さ方向にだけ伸びる（§9.2）。「応援する」もクリックで
  閉じ扱い（表示済みに記録）。トーストでは通知しない。言語切替時は
  `OnLanguageChanged` → `SetDonationBannerText()` で文言だけ差し替え。
- **§9.4 の隔離**: ポーリング経路からの呼び出し（`UpdateDonationBanner`）と設定保存・ブラウザ
  起動はすべて try/catch で包み、カンパ側の例外が監視機能に波及しない。
- 常設導線: 右クリックメニュー（ウィンドウ・トレイ共通）「説明」の下の「☕ 開発を応援」と、
  ヘルプウィンドウ末尾の Hyperlink の2つだけ。

## 多言語対応（実装済み、v0.3追補 §11）

`Resources/Strings.ja.xaml` / `Strings.en.xaml`（`sys:String` の ResourceDictionary）+
`Services/LocalizationService` の2層構成。**キー集合は ja/en で常に一致させること** —
`LocalizationDictionaryTests` がキー集合の一致・空値なし・`{n}` プレースホルダ集合の一致を検証して
いるので、キーを片方にだけ追加するとテストが落ちる。

- **XAML側**: 全て `{DynamicResource キー}` 参照。`LocalizationService.ApplyLanguage` が
  `Application.Current.Resources.MergedDictionaries` の辞書を差し替えるので再起動なしで即反映。
- **コード側**: `LocalizationService.T(key)` / `F(key, args)`（`string.Format`、CurrentCulture）。
  動的に組み立てる文字列（メニュー・トレイツールチップ・トースト・監視バーのラベル・
  RTK/ccusage表示行・インストール手順）は全てここ経由。切替時は `LanguageChanged` イベントで
  `MainWindow.OnLanguageChanged` が `RefreshHealth()`＋外部ツール再ポーリングを行い再生成する
  （メニューは毎回開くたびに再構築されるので対応不要）。
- **言語解決（§11.4）**: `NormalizeLanguage`（純粋関数・テスト済み）— 保存値 "ja"/"en" 優先、
  キーなし（初回）は OS の UI カルチャ、**不正値は en**（OSが日本語でも）。初回解決値は
  settings.json の `language` に保存される。未翻訳キーは en → キー名の順でフォールバックし、
  Debug ログにのみ警告（§11.5）。
- **Gotcha（初期化順序）**: `ExternalToolsHealth` の「計測中...」のようなフィールド初期化子で
  `T()` を呼んではいけない — MainWindow のフィールド初期化はコンストラクタ本文（＝
  `ApplyLanguage`）より先に走り、辞書ロード前なのでキー名がそのまま表示される。初期表示文字列は
  `ApplyLanguage` 後にコンストラクタ本文で代入する。
- **テストの流儀**: 単体テストは WPF `Application` なしで動くよう、`TestLocalization` が
  Strings.*.xaml を**素のXMLとしてパース**して `ApplyDictionaries` で注入する（pack URI は使わない）。
  言語状態はプロセスグローバルなので、**テストは全クラス ja 固定**（`TestLocalization.UseJapanese()`
  をコンストラクタで呼ぶ）。別言語に切り替えるテストを書くと xUnit の並列実行でレースするので禁止。
  表示文字列を検証する既存テスト（monitor/parser/display builder）はこの初期化が必須。

## 外見・お守りモチーフ（実装済み、v0.3追補2 §12）

一元管理の構造（どこに何があるか）:

- **色値の実体は `OmamoriTheme.cs` の `*Hex` 定数**（常盤 `#1B813E` / 琥珀 `#CA7A2C` / 紅緋
  `#C73E3A` / 金茶 `#B98C4A` / 生成り `#F8F4E6`）。`Resources/Theme.xaml` は `x:Static` でこれを
  §12.2 のキー名（`OmamoriHealthy`/`OmamoriWarning`/`OmamoriDanger`/`OmamoriAccent`/
  `WashiBackground`）の Brush として XAML に公開するだけ。**Theme.xaml を直接の色定義にしなかった
  のは意図的** — `UsageFractionToBrushConverter` のテストは WPF `Application` なし（pack URI 不可）
  で走るため、色の実体は素の C# に置く必要がある（§12.7 のテスト維持）。色を足す/変えるときは
  OmamoriTheme.cs に書き、Theme.xaml で公開し、**アイコンを再生成**する。
- `OmamoriDanger`（90%以上の赤）は §12.2 の表には無いが、バーの既存3段階（緑/琥珀/紅）を
  維持するために追加したキー。トレイ・ミニお守りは仕様どおり2状態。
- **図形の実体は `OmamoriShape.cs` の `*PathData` 定数**（24×24 座標系、pouch=袋 / loop=紐の輪 /
  knot=叶結び簡略）。ウィンドウ内ミニお守り（XAML `Path`）と生成スクリプトの両方がここを参照。
- **`tools/generate-icons.ps1`** が OmamoriShape.cs / OmamoriTheme.cs から正規表現で値を抽出し、
  WPF 描画で `TokenOmamoriTool/Assets/omamori.ico`（緑、アプリ兼トレイ正常）と
  `omamori-warning.ico`（琥珀、トレイ警告）を生成する（16/32/48/256、16px は knot 省略）。
  外部ツール不要（Windows PowerShell 5.1）。図形・色を変えたら再実行して .ico をコミット:
  `powershell -ExecutionPolicy Bypass -File tools\generate-icons.ps1`（`-PreviewDir` で確認用 PNG
  も出せる）。

ハマりどころ（実機で踏んだもの）:

- **`GeneratedIconSource` は任意図形を描けない**（文字＋楕円/矩形背景のみ）。このため §12.3 は
  「共有図形から生成した .ico の URI 参照2状態切替」に改訂済みで、実装は pack URI 参照の
  `BitmapImage`（.ico）を `IconSource` に渡す — これはトレイ常駐メモに記載の「URI-backed なら
  安全」経路。`RenderTargetBitmap` 直接代入は今も禁止（非同期クラッシュ）。
- **generate-icons.ps1 は UTF-8 BOM 必須**（日本語コメント入り .ps1 を PowerShell 5.1 は BOM
  なしだと ANSI として読みパースエラーになる）。編集時にエンコーディングを落とさないこと。
- PowerShell は関数戻り値の `byte[]` を object[] に展開するので、バイト列を返す関数は
  `return ,$bytes` にしてある。外すと .ico のディレクトリだけ書かれて画像データが消え、
  ビルドが `CS7065`（Win32リソース読み込み失敗）で落ちる。
- .ico のエントリは 16/32/48=BMP DIB（BITMAPINFOHEADER、高さ2倍、ANDマスク付き）、256=PNG の
  古典構成。全部PNGにすると古いシェル/デコーダで問題が出るため。
- **小窓の角丸はウィンドウ自体ではなく内側 Border**（CornerRadius=6、金茶の縁）。§12.6 が
  透過・変形ウィンドウを禁止しているため、Win10 の四角い窓のまま内側で丸みを出す設計。
- タイトル部のミニお守り（`MiniOmamoriPouch`）はウィンドウ内ヘッダー行にあり、
  `UpdateTrayState()` がトレイと同じ warning 判定で塗りを切り替える。ウィンドウ左上
  （タイトルバー）の `Icon` は §12.4 どおり緑固定 — 状態連動はウィンドウ内側のみ。
- csproj: `ApplicationIcon` が EXE 用、`<Resource Include="Assets\*.ico">` が pack URI 用。
  両方必要（どちらか片方では タスクバー/トレイ のどちらかが欠ける）。

## EXE化・配布（実装済み、v0.3追補3 §13）

- **発行は `tools/publish.ps1`**（Windows PowerShell 5.1、日本語コメント入りなので UTF-8 BOM
  必須）。csproj の `<Version>` を読んで `publish/TokenOmamoriTool-vX.Y.Z-win-x64/`（発行出力、
  スモークテスト用）・`.zip`・`.zip.sha256.txt` を出力する。§13.2 の警告どおり
  **PublishTrimmed / ReadyToRun は絶対に足さない**（WPF非対応）。
- **単一ファイルにならないのは仕様**: `PublishSingleFile=true` +
  `IncludeNativeLibrariesForSelfExtraction=true` でも、WPF のネイティブ DLL 5本
  （wpfgfx/PresentationNative/D3DCompiler/PenImc/vcruntime140 の `*_cor3.dll`）は .NET の制約で
  バンドルできず EXE の隣に出る。ZIP にはこの6ファイル全部を入れる（EXE 単体を配って
  起動しない、が典型事故）。EXE 約170MB / ZIP 約69MB。
- **バージョン表示**: csproj `<Version>`（git タグ・ZIP名と一致させる、§13.3）→ アセンブリ
  メタデータ → `Services/AppVersionInfo`（純粋関数、単体テスト済み）→ ヘルプウィンドウ末尾。
  ハードコード禁止。4番目のリビジョン部は表示で落とす（1.0.0.0 → 1.0.0）。
  **XAML の sys:String は末尾スペースがトリムされる**ので「バージョン: 」のような
  プレフィックスは作らない — 区切りスペースはコード側（HelpWindow）で足している。
- **スモークテスト実績（v1.0.0、発行版EXEで確認済み）**: 起動・ウィンドウ/タイトルバーの
  お守りアイコン・和紙テーマ、settings.json が EXE 隣に生成、初回 ja 自動判定、en 切替
  （pack URI 解決）、ヘルプのバージョン表示、×→トレイ格納（プロセス常駐）、メニュー「終了」で
  完全終了。トレイアイコンからの復帰はクリック自動化が不安定なため手動確認項目のまま。
- リリースノートは `docs/release_note_template.md`（日本語先・English後）。SHA256 は
  publish.ps1 の出力を転記。
